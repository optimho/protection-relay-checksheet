This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Purpose

This repository is a documentation system for routine and maintenance testing of protection relays. It produces two paired deliverables per relay:

- **Check sheet** (`.docx`) — printed/electronic procedure followed during the test visit; filled in by the tester.
- **Results template** (`.xlsx`) — live workbook capturing measured values, with auto pass/fail calculation and a rolled-up overall outcome.

The **Specification** (`Protection_Relay_Test_Documentation_Specification.docx`) is the authoritative structural reference. The **Onboarding Form** (`New_Relay_Onboarding_Form.xlsx`) captures per-relay parameters and drives generation of new pairs.

## Biennial (2-Yearly) Visual Check Sheets

Lighter-weight check sheets for routine in-service condition monitoring. Four files:

| File | Covers | Tabs |
|------|--------|------|
| `SEL_Biennial_Visual_Check.xlsx` | SEL-300G, SEL-311L, SEL-787, SEL-700G | 1 (generic SEL form) |
| `HIT_670_Biennial_Visual_Check.xlsx` | REG670, RET670 | 1 (Hitachi 670-series form) |
| `ABB_615_Biennial_Visual_Check.xlsx` | REF615, RET615, REM615, REU615 | 1 (615-series form) |
| `GE-889_Biennial_Visual_Check.xlsx` | GE Multilin 889 | 1 (889-specific form) |

Note the 670 biennial form is **Hitachi**-branded (`HIT_670…`, title "Hitachi RELION 670") since the Relion 670 line is now a Hitachi Energy product; the 615 form stays **ABB**-branded. The DCS comms check still references **ABB 800XA** in both, as the DCS itself is unchanged. (The 670 results templates keep their `REG670_…` / `RET670_…` filenames — the model numbers are unchanged — but are likewise Hitachi-branded.)

The 670 and 615 forms are split into separate files because their front-panel paths, self-supervision signals, and measurement labels differ significantly (LHMI graphical vs text-based, INTERRSIG vs IED status, IA/IB/IC vs IL1/IL2/IL3). The GE 889 uses EnerVista software and Modbus TCP — no serial commands; all navigation is via front panel menus or EnerVista.

Sections per check: connection method instructions → self-test status → primary-value measurements → clock check → 12-row event log → DCS/Modbus TCP comms check → optional record clear → sign-off.

Generated by: `generate_SEL_biennial_check.py`, `generate_HIT670_biennial_check.py`, `generate_ABB615_biennial_check.py`, `generate_GE889_biennial_check.py`

## File Layout

```
Doc files/          ← reference Word documents (spec, workflow, SEL check sheets)
*.xlsx              ← all maintenance templates + biennial check sheets
generate_*.py       ← one Python script per relay type (openpyxl/python-docx); tracked in git
add_branding.py     ← post-processor: applies logo + footer to all xlsx
.gitignore          ← excludes *.pdf (manuals) and .~lock.* (LibreOffice locks)
```

**Caution — scripts are *origin* generators, not live source-of-truth.** Several xlsx have
been hand-edited after generation (the "v1 canonical" versions), so re-running a
`generate_*.py` can produce a *different* sheet and overwrite those edits. Verified example:
`generate_HIT670_biennial_check.py` no longer reproduces `HIT_670_Biennial_Visual_Check.xlsx`
(the live file was hand-extended). Before regenerating any template, diff against the live
file first, or treat the xlsx as canonical and edit it directly.

## Workflow for Creating a New Relay Pair

See `Doc files/workflow for creating a checksheet.docx` for the full workflow. Recommended approach:

1. Fill in `New_Relay_Onboarding_Form.xlsx` for the new relay (ratios, element list, binary I/O labels, relay-specific blocking checks).
2. Provide the Specification, the filled Onboarding Form, and the closest existing pair (SEL-311L for line relays; SEL-300G for generator relays) to an LLM.
3. Prompt: *"Generate the `<Relay>_Test_Check_Sheet.docx` and `<Relay>_Test_Results_Template.xlsx` for `<relay model>` following the attached Specification, using the Onboarding Form parameters and the provided pair as a worked example."*
4. Review and iterate.
5. Run `python3 add_branding.py` to apply Contact Energy logo and footer.

## Document Structure

### Check Sheet (.docx) — fixed section order

| Section | Content | Varies per relay? |
|---------|---------|-------------------|
| Cover | Logo, title, relay subtitle | Subtitle only |
| 1 — Test Information | Site, personnel, relay under test (CT/VT ratios), equipment, drawings, SharePoint links | Ratio rows in 1.3 |
| 2 — Safety & Permits | Standard LOTO checklist + isolation record | Append relay-specific blocking checks |
| 3 — Pre-Test Checks | Comms, status, clock, event records, SOE, as-found settings, settings vs PFD | No |
| 4 — Analogue Input Verification | Primary-value injection tables (one sub-section per CT/VT group) | Yes — sub-sections vary |
| 5 — Binary I/O | Input pickup + output continuity tables | Prefilled function labels vary |
| 6 — Protection Element Testing | Omicron test structure, element tracker, per-element result template | Element list varies |
| 7 — Post-Test Checks | Re-compare settings, clear events, time sync, restore plant | Restoration items vary |
| 8–10 — Findings, Files Index, Sign-Off | Defect log, artefact index, tester/witness/engineer sign-off | No |

**Key design principle (Section 4):** Always verify the relay reports the correct **PRIMARY** value (injected secondary × ratio), not just the secondary value. This catches ratio-setting errors that secondary-only checks miss.

### Results Template (.xlsx) — 9 tabs

| Tab | Mirrors section | Live calculation |
|-----|----------------|-----------------|
| 1. Cover | Front matter + relay info | Ratios drive analogue formulas; tolerances drive P/F; auto roll-up |
| 2. Equip & Refs | Equipment + drawings + locations | Clickable hyperlinks |
| 3. Pre-Test | Section 3 | OK/FAIL conditional formatting |
| 4. Analogue Inputs | Section 4 | Expected Primary, Mag Err %, Angle Err°, P/F — all formulas |
| 5. Binary IO | Section 5 | PASS/FAIL conditional formatting |
| 6. Element Tracker | Section 6.3 | PASS/FAIL |
| 7. Element Results | Section 6.4 | Pickup err %, Time err %, Result — formulas |
| 8. Post-Test | Section 7 | OK/FAIL |
| 9. Files & Sign-Off | Sections 8–10 | Overall result pulled from Cover |

### Formula Architecture (xlsx)

**Layer 1 — Defined names on Cover sheet** (workbook-scoped, readable names):

Per-relay name conventions (all workbook-scoped):

| Template | CT ratio names | VT ratio names |
|----------|---------------|----------------|
| SEL-787 | `Ratio_CTR1`, `Ratio_CTR2`, `Ratio_CTRN` | `Ratio_PTR` |
| SEL-700G | `Ratio_CTRX`, `Ratio_CTRY`, `Ratio_CTRN` | `Ratio_PTRX` |
| REG670 | `Ratio_CT_stator`, `Ratio_CT_Y`, `Ratio_CT_neut` | `Ratio_VT_main`, `Ratio_VT_neut` |
| RET670 | `Ratio_CT_W1`, `Ratio_CT_W2`, `Ratio_CT_W3` | `Ratio_VT` |

Tolerance names are identical across all templates: `TolV_pct`, `TolI_pct`, `TolAngle_deg`, `TolPickup_pct`, `TolTime_pct` → yellow fill cells on Cover.

**VT ratio formula:** `VTprim_kV * 1000 / VTsec_V` (primary stored in kV for high-voltage relays; secondary in V). Neutral VTs follow the same convention.

**Layer 2 — Per-row in Analogue Inputs tab:**
- Expected Primary = `injected_secondary × Ratio_<applicable>`
- Mag Err % = `(Relay Primary − Expected Primary) / Expected Primary × 100`
- Angle Err° = `Relay Angle − Injected Angle`
- P/F = `IF(OR(ABS(mag_err) > TolV_pct, ABS(angle_err) > TolAngle_deg), "FAIL", "PASS")`
- All formulas wrapped in `IFERROR` + blank-input guards (no `#DIV/0!` on empty rows)

**Layer 3 — Cover roll-up:**
- Analogue: `COUNTIF` for FAIL/PASS across Analogue Inputs tab
- Elements: `COUNTIF` across Element Results tab
- Auto overall: FAIL if either is FAIL; PASS if both PASS; else INCOMPLETE
- The **manual dropdown** (PASS / PASS WITH NOTES / FAIL) on Cover is the authoritative signed result; auto overall is a cross-check only

## Key Differences Between Relay Families

### SEL relay family (all models)
- Serial ASCII protocol: `STA` (self-test), `METER` (primary values), `HIS`/`EVE`/`SER` (events), `TIM`/`DAT` (clock)
- Settings tool: AcSELerator QuickSet (.sel files)
- CT ratios stored as `CTR×` (primary/secondary, ratio stored as integer primary amps, secondary 1 or 5 A)
- VT ratio: `PTR×` (primary kV × 1000 / secondary V, stored as primary V)
- Self-test: `STA` command → HALARM (hardware) and SALARM (software) bits; issue `STA C` to clear warnings
- Front panel: `MAIN > Meters > Fundamental` for primary values; `MAIN > Targets > Row 49` for digital inputs (700G), `Row 21` (787)

### Hitachi Relion 670 family (REG670, RET670)
- Tool: PCM600 (.pcm project file); LHMI graphical touchscreen
- CT settings per winding: `CTprimN` / `CTsecN` (primary A / secondary A)
- VT settings: `VTprimN` (kV) / `VTsecN` (V) — note kV not V for primary
- Self-supervision: `INTERRSIG` function block via `LHMI → Diagnostics → IED status → Signals`
  - Signals: FAIL, WARNING, RTCERROR, TIMESYNCHERROR, RTEERROR, IEC61850ERROR, WATCHDOG, LMDERROR, APPERROR
  - Hardware modules: PSM, ADOne/Two, BIM, BOM, IOM, MIM, LDCM
  - INTERNAL FAIL contact on PSM asserts when FAIL signal is active
- Measurements: `LHMI → Measurements → [function block]` — all in primary units
- Events: `LHMI → Events → Disturbance records`
- DCS: IEC 61850 GOOSE/MMS to ABB 800XA
- Differential balance: IOP/IRT in pu — accept IOP ≤ 0.05 pu under load (tighter than SEL 0.10)

### ABB Relion 615 family (REF/RET/REM/REU615)
- Tool: PCM600 or web browser; LHMI is simpler text-based 4-line display
- Status LEDs: READY (green solid = healthy), START (yellow), TRIP (red)
- Self-supervision: `LHMI → Monitoring → Self-supervision → IED status`
- Measurements: `LHMI → Monitoring → Measurements → Current/Voltage`
- Events: `LHMI → Events → Event list`
- Current labels: IL1/IL2/IL3 (not IA/IB/IC like SEL/670)
